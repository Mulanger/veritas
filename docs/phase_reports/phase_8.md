# Phase 8 Completion Report

**Date:** 2026-04-26  
**Branch:** `phase/8-audio-detector`  
**Status:** Complete

## Summary

Phase 8 adds an on-device audio detector path for standalone audio files. Audio now decodes through Android `MediaExtractor`/`MediaCodec`, normalizes to 16 kHz mono waveform input, runs a signed wav2vec2-base TFLite model through Play Services LiteRT, fuses the model score with codec sanity checks, and returns calibrated `DetectorResult`/`Verdict` data through the existing provenance pipeline.

## Final Model

- Runtime model: `Hemgg/Deepfake-audio-detection`
- Hugging Face revision: `0d75271368ef2c7efd14831dc503c431f6aab0eb`
- License: Apache 2.0
- Base architecture: `facebook/wav2vec2-base`
- Output label order: `logit[0] = AIVoice`, `logit[1] = HumanVoice`
- Runtime asset: `data-detection-ml/src/main/assets/models/audio/deepfake-audio-detector-hemgg-wi8.tflite`
- Quantization: ai-edge-quantizer weight-only INT8 (`wi8_afp32`) with internal TFLite buffers
- Final TFLite size: `96,190,928` bytes
- SHA-256: `3046375262e631f25eb801c3480306235731f9bd95cedcf450663a31116f0b4c`
- Signature: Ed25519, same Phase 7 public key
- ONNX parity: softmax MAE `0.02847679`
- Tensor dtype distribution: `1245 float32`, `83 int8`, `27 int32`

The original `as1605/Deepfake-audio-detection-V2` path was rejected because converted artifacts either exceeded the `120 MB` cap or failed runtime invocation. Hemgg became viable only after forcing ai-edge-quantizer to serialize internal flatbuffer buffers; the default external-buffer output failed on Android with `Input tensor 255 lacks data`.

## Dataset

The local audio golden set is generated by `tools/eval-datasets/build_audio_golden_set.py`.

- Manifest: `feature-detect-audio/src/androidTest/assets/golden-audio/MANIFEST.csv`
- Audio payload: ignored by git under `golden-audio/real/` and `golden-audio/synthetic/`
- Real sources: Common Voice English and MINDS14 English clips already pulled for calibrated conversion
- Synthetic source: local Windows SAPI TTS clips already generated for calibrated conversion
- Codec variants: WAV, MP3, M4A, Opus

This is product-functional coverage, not a production-quality audio benchmark. The weakest bucket is Opus, which should be prioritized in the retraining/eval workstream alongside ASVspoof, WaveFake, and current commercial TTS samples.

## Runtime Integration

- New module: `feature-detect-audio`
- Shared runtime addition: `data-detection-ml/preprocessing/AudioPreprocessor.kt`
- Pipeline integration: `ProvenancePipeline` routes `MediaType.AUDIO` through `AudioDetector` after C2PA/SynthID pre-flight checks
- Sub-score keys: `wav2vec2_model`, `codec`
- Fusion weights: `0.92 wav2vec2_model / 0.08 codec_signal`
- Audio model execution: CPU XNNPACK only for Phase 8

CPU-only execution is intentional for this model. The weight-only wav2vec2 graph passed the p95 latency gate on CPU, and NNAPI remains forbidden by D-016.

## Demo Script

1. Build the golden audio set:
   `py -3 tools/eval-datasets/build_audio_golden_set.py`
2. Use any `feature-detect-audio/src/androidTest/assets/golden-audio/synthetic/*.wav` or `*.mp3` file as the synthetic voice demo clip.
3. Use any `feature-detect-audio/src/androidTest/assets/golden-audio/real/*.wav` or `*.mp3` file as the real speech demo clip.
4. Share the file to Veritas or scan it through the file-picker path.
5. Synthetic samples should lean `LIKELY_SYNTHETIC` unless uncertainty gates fire; real samples should lean `LIKELY_AUTHENTIC` or `UNCERTAIN` for low-quality/Opus-heavy cases.

## Known Limitations

- Hemgg is a public wav2vec2-base detector; it is not calibrated against current 2026 commercial voice clones.
- The local eval set uses SAPI TTS rather than ASVspoof/WaveFake/current commercial TTS, so reported quality is a product-functional gate, not a launch benchmark.
- Opus is the weakest codec bucket in the current eval.
- Full-integer INT8 was not viable for this graph in the current Windows conversion environment; shipped quantization is weight-only INT8 with float32 activations.
- Bundling both Phase 7 and Phase 8 models increases APK size; D-041 already assigns model delivery/app-size mitigation to Phase 13.
